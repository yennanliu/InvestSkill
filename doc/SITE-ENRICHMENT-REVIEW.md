# InvestSkill Site Review & Enrichment Proposal

> **Scope:** Review of the deployed documentation site ([yennj12.js.org/InvestSkill](https://yennj12.js.org/InvestSkill/), formerly yennanliu.github.io/InvestSkill) and ideas for enriching it with more concepts, knowledge, and use cases.
> **Date:** 2026-06-15
> **Nature:** Review and recommendations only — no files were changed.

---

## 1. What the Site Is Today

The site is generated by `docs/build-site.js`, which renders a fixed set of repo markdown files into a dark-themed, three-pane docs UI (sidebar nav · content · on-this-page TOC). The published pages are:

| Page | Source file | Purpose |
|------|-------------|---------|
| Quick Start (home) | `README.md` | What it is, install, 21-framework table, workflows |
| Cookbook | `COOKBOOK.md` | Setup, core concepts, 8 demo outputs, 6 workflows, cross-AI usage, tips |
| Contributing | `CONTRIBUTING.md` | How to contribute |
| Changelog | `CHANGELOG.md` | Release history |
| 繁體中文 / 操作手冊 | `README-zh-TW.md` / `COOKBOOK-zh-TW.md` | Chinese mirrors |

**Strengths.** Clean UX (search, theme toggle, copy-markdown, mobile sidebar). Strong "what each skill produces" coverage. Good cross-platform story (Claude Code, Cursor, Gemini, Copilot, ChatGPT). The Cookbook's demo outputs and end-to-end workflows are the best assets — they show, not just tell.

**Core gap.** The site explains *what each of the 21 skills does* and *how to run them*, but barely teaches *the investing knowledge behind them* or *how to interpret what comes back*. Terms like Piotroski F-Score, ROIC, WACC, days-to-cover, and IV rank appear with no definition. There is no glossary, no concept primer, no "which skill when" decision guide, and no documentation of where data comes from or how to trust an AI-generated number. For a tool whose whole value is *teaching an AI to think like an analyst*, the site under-teaches the human.

---

## 2. Gaps at a Glance

| # | Gap | Impact |
|---|-----|--------|
| 1 | No glossary of financial terms | Beginners bounce off jargon; metrics feel like black boxes |
| 2 | No concept primers (the "why" behind each metric) | Users run skills without understanding output |
| 3 | No "which skill when" decision guide | 21 skills is overwhelming; no map from goal → skill |
| 4 | No interpretation guides (what a score *means* / what to do) | Output is produced but not actionable for novices |
| 5 | No data-sourcing / accuracy / hallucination guidance | Trust problem — users can't tell good output from confident-wrong output |
| 6 | Thin persona/use-case coverage | One implied user (a savvy analyst); no beginner, dividend, swing, or macro paths |
| 7 | No worked "anti-patterns" / failure examples | Users repeat avoidable mistakes |
| 8 | No per-skill reference pages on the site | The 21 `prompts/*.md` aren't surfaced as browsable pages |
| 9 | No visual learning aids (decision trees, metric maps) | All prose/tables; concepts that are inherently visual stay flat |
| 10 | No "responsible use / limitations" depth beyond a one-line disclaimer | Risk and compliance posture is underdeveloped |

## 3. Enrichment Ideas

Ideas are grouped by theme. Each notes **what to add**, **why it helps**, and **where it fits** in the existing build (most are new markdown files + a few `NAV_SECTIONS`/`PAGES` entries in `docs/build-site.js` — no architecture change).

### Theme A — Concepts & Knowledge (the biggest lever)

**A1. Financial Glossary (`GLOSSARY.md` → new "Reference" page).**
A–Z of every term the skills emit: Piotroski F-Score, ROIC/ROE/ROA, WACC, FCF, EV/EBITDA, P/E vs P/S vs PEG, moat, beta, days-to-cover, short float, IV rank, the Greeks, dividend payout/coverage ratio, Porter's Five Forces, residual income, terminal value. Each entry: one-sentence definition, the formula, a "good/bad range," and which skill surfaces it. Cross-link terms to where they appear in the Cookbook. *Highest-value, lowest-risk addition.*

**A2. Concept Primers / "Investing 101" track (`CONCEPTS.md` → new "Learn" section).**
Short explainers tying metrics into mental models, not isolated definitions:
- *Quality vs. Value vs. Growth* — the three lenses InvestSkill blends.
- *How intrinsic value works* — DCF intuition before the formula (why discount rate and terminal value dominate).
- *Reading a moat* — what durable competitive advantage actually looks like in the numbers.
- *Margin of safety & position sizing* — turning a signal into a decision.
- *Risk types* — market, idiosyncratic, liquidity, concentration, rate sensitivity.
- *Macro regimes* — how rates/inflation steer sector rotation (ties to `economics-analysis` + `sector-analysis`).

**A3. "Anatomy of the Signal Block" deep-dive.**
The signal block is the product's signature output, but the site only shows the box. Add a page that dissects each field (Signal vs. Confidence vs. Conviction vs. Horizon — they're different axes people conflate), explains the 0–10 score bands with examples of what pushes a score up/down, and shows how the composite block in `research-bundle`/`full-report` is assembled from sub-signals.

### Theme B — Navigation & Decision Support

**B1. "Which Skill Should I Use?" decision guide (`CHOOSE-A-SKILL.md`).**
A goal-first map: *"I want to… screen quickly → `stock-eval`; know if it's cheap → `stock-valuation`/`dcf-valuation`; check if the dividend is safe → `dividend-analysis`; time an entry → `technical-analysis`; understand the business → `competitor-analysis` + `fundamental-analysis`."* Pair with a one-screen decision tree (Mermaid renders in the current pipeline).

**B2. Skill comparison guides.**
Short "X vs Y, and when to use each" pages users actually search for:
- `stock-eval` vs `fundamental-analysis` vs `stock-valuation` (overlap is real and confusing).
- `dcf-valuation` vs `stock-valuation` (single-method vs multi-method).
- `research-bundle` vs `full-report` (synthesis-in-chat vs exported HTML).
- Fundamental vs technical — when each is the right tool.

**B3. Per-skill reference pages on the site.**
The 21 `prompts/*.md` files are the actual frameworks but aren't browsable on the site (only via GitHub raw). Generate one HTML page per skill (the build loop already iterates a `PAGES` array — extend it to glob `prompts/`). Each page: purpose, inputs, sample output, the metrics it emits (linked to the glossary), and "pairs well with."

### Theme C — Use Cases & Personas

**C1. Persona-driven journeys (`USE-CASES.md`).**
The Cookbook has 6 workflows but they assume one user type. Add goal-oriented end-to-end stories:
- *The dividend/income investor* — build & stress-test an income portfolio (`dividend-analysis` → `portfolio-review`).
- *The first-time investor* — "I have $5k and want to evaluate my first stock" (`stock-eval` → glossary links → `result-validator`).
- *The earnings-season trader* — pre/post-earnings playbook (already partly in Cookbook; expand into a narrative).
- *The macro/top-down allocator* — regime → sector → name (`economics-analysis` → `sector-analysis` → `stock-eval`).
- *The skeptic / risk manager* — using `result-validator`, `insider-trading`, `short-interest` to find reasons *not* to buy.

**C2. Worked case studies with full reasoning.**
Pick 2–3 real tickers and walk the *entire* thought process end to end — including where the analysis changed the conclusion (e.g., "fundamentals looked great until the 10-K red flags and insider selling flipped it to HOLD"). The current demos show output format; case studies would show *judgment*.

**C3. Anti-patterns / "How analyses go wrong."**
Short page: over-trusting a single DCF, ignoring confidence level, using stale data, mistaking a yield trap for income, anchoring on one signal block. Each with the fix.

### Theme D — Trust, Data & Responsible Use

**D1. "Data & Accuracy" page.**
Be explicit: skills don't fetch data themselves — they reason over what the user provides or what the host AI can access. Document recommended data sources, how to paste real financials, why outputs vary by model, and how to spot hallucinated numbers. This directly addresses the trust gap.

**D2. "Validating AI Output" expansion.**
Promote `result-validator` from a tip to a first-class workflow: when to run it, how to read its confidence score, and a manual sanity-check checklist (do the numbers reconcile? is the data current? does the signal match the narrative?).

**D3. Expanded limitations & responsible-use section.**
Beyond the one-line disclaimer: not financial advice, model-dependence, no real-time data guarantee, the importance of primary sources, and that signal blocks are decision *aids*, not decisions.

### Theme E — Presentation & Discoverability

- **E1. Glossary tooltips / cross-links** — link jargon in the Cookbook to glossary anchors.
- **E2. Visual aids** — decision trees, a metric-relationship map, a "skill dependency" diagram (all Mermaid, already supported).
- **E3. A real search index** — the ⌘K box is currently a placeholder; a small client-side index (e.g. lunr/Pagefind over the built pages) would make the growing content findable.
- **E4. Printable one-page "cheat sheet"** — all signal-score bands, key formulas, and the skill-picker tree on a single page.
- **E5. Surface a learning path** — a "Start here → Concepts → Choose a skill → Worked case → Validate" ordered track for newcomers.

## 4. Suggested Priority

| Priority | Items | Rationale |
|----------|-------|-----------|
| **P0 — quick wins, high value** | A1 Glossary, B1 Skill-picker, D1 Data & Accuracy | Directly close the worst gaps; pure markdown, no build changes beyond a nav entry |
| **P1 — depth** | A2 Concept primers, A3 Signal-block deep-dive, C1 Persona journeys, D2 Validation | Turn "what" into "why + how to act"; build on existing Cookbook strengths |
| **P2 — scale & polish** | B3 Per-skill pages, C2 Case studies, E2 Visual aids, E3 Search index | Higher effort; benefit grows as content grows |
| **P3 — nice to have** | B2 Comparison guides, C3 Anti-patterns, D3 Limitations, E4 Cheat sheet, E5 Learning path | Rounds out the experience |

## 5. Implementation Notes (no code written here)

All P0/P1 ideas fit the current build with minimal effort:

1. Add new `.md` files at repo root (consistent with how `README.md`/`COOKBOOK.md` are sourced).
2. Register each in the `PAGES` array and `NAV_SECTIONS` in `docs/build-site.js` — consider a new **"Learn"** nav section (Concepts, Glossary, Choose a Skill) and a **"Trust"** grouping.
3. For per-skill pages (B3), extend the build loop to iterate `prompts/*.md` so the 21 frameworks become first-class site pages.
4. Mirror new pages into Chinese to keep parity with the existing `zh-TW` track (can lag the English).
5. Keep the SKILL.md ↔ prompts/*.md sync rule in mind if any concept content is embedded in skills.

None of the above requires changing the plugin architecture, the signal-block contract, or the test suite's structural checks — they are additive content pages.
